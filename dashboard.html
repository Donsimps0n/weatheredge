How WeatherEdge Finds Edge

The complete pipeline behind every trade signal

WeatherEdge automates the entire Polymarket weather trading pipeline — from market discovery to trade execution — using real GFS ensemble forecasts and fee-adjusted Kelly sizing.

01

Market Discovery

Queries Polymarket's Gamma API using direct slug-based lookup for all 23 cities next 7 days. Falls back to tag search for any non-standard slugs. Identifies all active temperature bins per event.

02

Fetch GFS 30-Member Ensemble

Downloads 30 independent GFS model runs from Open-Meteo at ICAO airport station coordinates (not city centres — airport stations match Polymarket's resolution source). Run 4 daily at 00z/06z/12z/18z UTC.

03

KDE Probability Calculation

Uses Gaussian KDE (Scott's bandwidth) over 30 ensemble samples to compute P(temperature falls in bin). Bayesian Beta-Binomial smoothing corrects for small sample uncertainty. Returns probability + uncertainty estimate.

04

NWS Observation Blend (US only)

Once the day has started, blends real NWS station observations with remaining forecast hours. obs_weight = hour_now/24. Reduces uncertainty as more of the day is measured.

05

Edge Detection & Dual-Source Confirmation

edge = GFS_probability → market_price. Only trades where both GFS ensemble AND NWS deterministic forecast agree on the direction (within 5F/3C). Eliminates model-conflict situations like Dallas on March 20 (GFS 88F vs NWS 94F skip).

06

Fee-Adjusted Kelly Sizing

f* = (p×W − q×L) / (W×L) / (WL) where W = payout → fee, L = 1 + fee. Uses quarter-Kelly (0.25) by default. Discounts Kelly by uncertainty estimate and lead-day skill decay. LMSR price impact check prevents over-sized orders.
